## 2. 技术实现深度描述 (Detailed Implementation)

### 2.1 核心基石：2-of-2 MPC 密钥切片与“对称性破解”
为了实现“双向不信任”的安全承诺，ATA 协议在 TLS 握手的底层进行了重构：
*   **非对称密钥碎片**：在 DHE 密钥交换过程中，插件端（Prover）生成随机数 $r_1$，公证方（Notary）生成 $r_2$。
*   **联合生成密钥**：通过 MPC 算法，双方在互不告知分片的前提下，共同生成 TLS 会话的对称加密密钥（Session Key）。
*   **打破对称性安全 (Breaking Symmetry)**：在标准 TLS 中，由于加密是对称的（AES-GCM），拥有密钥的 Prover 可以随意伪造数据。但在 ATA 中，由于密钥是碎片化的，Prover 在没有 Notary 协助的情况下无法计算出正确的 **Auth Tag (MAC)**。这意味着：**任何能通过验证的密文流，在数学概率上必然、且只能来源于 Twitter 的服务器。** 这就是“Oracle-less”验证的公信力来源。

### 2.2 物理层突破：Social Miner（本地社交挖矿机）
解决浏览器沙盒物理限制的终极武器：
*   **Native Messaging 桥接**：插件通过原生消息管道与独立运行的 `miner.exe/app` 通信。
*   **TCP 发射与指纹模拟**：Miner App 拥有开启原生 TCP 端口的权限，能够精确模拟浏览器的 TLS 指纹（JA3），使该请求在 Twitter 看来完全是用户在真实浏览器中的原生操作。
*   **家庭 IP 出口 (Residential IP)**：利用用户本机的真实网络环境。由于 Twitter 无法封锁分散全球的真实家庭 IP，这彻底解决了 VPS 方案极易被 IP 封锁的痛点。
*   **三层连通架构 (Triple-Layer Connectivity)**：
    1.  **自发现层 (UPnP/NAT-PMP)**：本地矿机利用原生权限尝试开启路由端口映射，实现点对点直连。
    2.  **超级节点层 (P2P Relay)**：具备公网 IP 的节点被标记为“超级矿机”，负责为 NAT 后的节点提供中继流量，仅转发加密流而不参与解密。
    3.  **官方兜底层 (Official Fallback)**：作为全网第一个公证节点，官方服务器为系统提供冷启动支持与最后的连通保障。

### 2.3 协议交互逻辑：直连直通与异步镜像 (Secure Mirroring Architecture)
为了兼顾“零延迟体验”与“数据防伪”，ATA 协议采用了独特的镜像复制模型：

1.  **阶段一：信令撮合与 MPC 预握手 (Pre-Handshake)**：
    *   小明的 Miner (Prover) 触发行为时，通过信令服务器随机匹配小红的 Miner (Notary)。
    *   双方跨网络执行 MPC 握手，共同派生 TLS 会话密钥分片（各持 1/2）。此时，双方均未连接 Twitter。
2.  **阶段二：流量直连与加密流镜像 (Direct Connect & Encrypted Mirroring)**：
    *   **直连 Twitter**：小明的 Miner 模拟浏览器指纹，利用本地住宅 IP **直接连接** Twitter 服务器，发起加密请求。请求不经过小红，避免了代理带来的 IP 穿透失败与通讯延迟。
    *   **加密镜像 (Carbon Copy)**：小明在接收 Twitter 下行加密流量的同时，将原始加密字节流（密文）**实时异步镜像**一份给小红。
3.  **阶段三：异步见证与证明生成 (Async Attestation)**：
    *   **防伪校验**：小红利用手里的密钥分片校验镜像密文的 MAC (Auth Tag)。由于密文来自 Twitter 且 MAC 通过 MPC 绑定，小明无法修改密文内容。
    *   **按需披露**：任务完成后，小明提供特定 Offset 的解密分片。小红结合镜像密文和分片，还原明文事实（如 `followed_by: true`），并签发 EIP-712 证明。

**架构优势**：用户感知延迟为零（RTT 保持原生），且小红无需充当流量中转节点（Proxy-less），极大降低了带宽压力和 IP 封锁风险。

### 2.4 动态偏移量适配与上下文窗口验证 (Dynamic Offset & Context Window)
针对 Web2 接口（如 GraphQL）返回 JSON 结构不固定、Offset 随机化的问题，系统引入了自适应验证机制：
*   **上下文特征匹配 (Pattern Matching)**：Notary 不再信任绝对偏移量，而是解开包含目标字段及其“词法邻居”（如 `"followed_by":` , `"is_blue_verified":` 等 Key 值）的一个 **上下文滑动窗口 (Sliding Window)**。
*   **语义完整性校验**：通过解密窗口内的词法结构，Notary 可以确信提取的 Value 确实属于目标字段，而非 Prover 拼凑的虚假片段。这种“找邻居”的策略使系统对 API 结构的动态变化具备了极强的鲁棒性。

### 2.5 链上清算机制与智能合约解析 (On-chain Settlement & Contract Interpretation)
见证生成的 Proof 作为一份“通用收据”，其核心目标是实现“验证即结算”的无人干预流程：

1.  **结构化证明 (EIP-712 Standard)**：Notary 签发的证明遵循 **EIP-712** 强类型签名标准。它将“用户地址”、“任务指纹”、“社交事实结果”封装为合约可直接解析的结构化数据块。这确保了证明在链上是**类型安全**且**易于检索**的。
2.  **信任名录管理 (Notary Registry)**：智能合约维护一个受信任的公证人公钥列表（由 DAO 治理）。当证明提交时，合约通过内置的 `ecrecover` 逻辑即时验证签名人的身份。只有存在于 Registry 中的节点签发的证明才具备法律效力。
3.  **分层赞助路由 (Layered Gas Sponsorship)**：
    *   **L3 内部极速通道**：在 TweetCat L3 链上，采用“赞助节点 (Sponsor Nodes)”模式。验证节点或广告主合约直接承担 L3 的微额 Gas，用户在日常交互中完全实现 **Zero-Gas 体验**。
    *   **跨层结算与 x402 集成**：当资产需跨链至 Base (L2) 或主网时，系统自动激活 **x402 协议**。由协议中继器（Relayer）代付高额 Gas 费，并通过批量处理降低成本，确保用户在获取真实价值资产过程中的极致流畅性。
4.  **透明度与可审计性**：生成的证明存档于链上或去中心化存储中，供广告主实时查看与对账，彻底解决了传统社交广告中“虚假流量”与“归因黑盒”的问题。

### 2.6 通信安全与跨进程凭证注入 (Native Messaging & Secure Injection)
为了解决浏览器扩展无法直接操作 TCP 协议且 Cookie 受限的问题，系统采用了 Native Messaging 安全总线：
1. **取消本地端口 (No Port Logic)**：弃用不安全的本地 WebSocket (避免端口劫持)，采用 Chrome 原生消息管道，通过进程间标准流进行双向通信。这确保了 Miner App 仅能由 TweetyCat 插件调起和交互。
2. **特权提取与注入 (Privileged Injection)**：利用插件 Service Worker 的特权环境，在用户触发行为时，毫秒级提取 `HttpOnly` 属性的 `auth_token` 与 `ct0` 凭证，并通过安全总线注入本地 Miner App 的加密引擎。
3. **环境分离 (Environment Decoupling)**：Miner App 接收凭证后，在原生 Native 环境中完成 TLS 1.3 报文的封装与发射。这种架构成功绕过了 Web 端对跨域和底层协议的限制，同时为敏感数据建立了一道物理隔离墙，极大提升了系统的安全性与生存能力。

## 3. 终极进化：P2P 协同公证与“极简选择性解密”

我们不仅去中心化了 Prover，还要去中心化 Notary：
1.  **我为人人 (Relay Mining)**：每个安装了“社交挖矿机”的用户，在闲置时都可以充当他人请求的“公证节点”。作为回报，公证节点在每笔验证金中抽取 **“中继算力费用”**。
2.  **无需电路的选择性解密 (Selective Disclosure via CTR Masking)**：
    *   **原理**：TLS 1.3 的 AES-GCM 采用计数器模式（CTR）。Notary 验证完整密文的 MAC 后，Prover 仅揭露包含关键数据（如 `is_blue_verified`）的特定 Offset 计数器块。
    *   **盲态验证**：Notary 协助解密该特定片段，验证其 JSON 路径与值是否符合预期模板，而对于 JSON 中包含的其余敏感信息（如私聊内容、Cookie 详情），Notary 因无对应计数器块的参与而完全处于“盲态”。
    *   **高效率**：这种基于流式加密特性的局部揭露，相比生成庞大的 ZK Proof，执行效率提升了 100 倍以上。
3.  **抗审查性**：公证节点不再是固定的几台服务器，而是分布在全球的终端。没有任何机构能通过封锁特定节点来使系统瘫痪。
4.  **场景普适化：从单站验证到全球 Web2 事实预言机 (Scenario Generalization)**：
    *   **跨平台价值归因 (Universal Attribution)**：支持跨主流社交矩阵（如 X, YouTube）、流媒体平台（如 Spotify）及大型电商枢纽进行跨域行为审计。任何基于内容驱动的商业推介，均可转化为可验证的链上资产，实现全网范围内的“推介即收益”。
    *   **创意资产溯源公证 (IP & Content Notarization)**：通过截获内容发布瞬间的 TLS 握手凭证，为全球数字创作者提供基于“平台原始证据”的原创时间戳服务。这为数字资产在跨域流转中的版权确认提供了最底层的物理级证据。
    *   **泛商业共识结算 (Unified Commercial Incentive)**：品牌方可自由定义复杂的数据触发指纹（如特定数字内容 ID、完播率关键点、转化路径入口）。协议充当了 Web2 物理操作与 Web3 自动化清算之间的“万能翻译器”，构建起一个零摩擦的全球数字结算网。
5.  **语义完整性证明 (Semantic Integrity Proof)**：
    *   **核心论证**：由于 TLS 1.3 的 MAC (GCM Tag) 是对整个加密流的承诺。在 2-of-2 MPC 架构下，Prover 无法自主生成合法的 Tag。
    *   **抗伪造性**：当 Notary 解开包含特定“词法邻居”（如字段名）的上下文窗口时，这种特定的字节组合在密文流中的出现，在概率上必然意味着该片段与前后的 Session 状态是原子绑定的。攻击者无法在不掌握 Notary 所持分片的情况下，通过“剪切-粘贴”或其他方式在加密流中植入带有正确语义特征的虚假块。
5.  **公证人准入与抗合谋机制 (Notary Admission & Anti-Collusion)**：
    *   **节点注册 (Node Registry)**：只有在信令中心注册并公示公钥的节点才具备签名权限。合约内置白名单或质押机制，仅受信任的或有资产抵押的节点生成的签名才会被视为有效。
    *   **随机指派 (Random Assignment)**：Prover 无法自主选择公证者。系统通过信令层进行随机撮合，切断证明者与公证者之间的合谋链路。
    *   **经济约束 (Economic Security)**：引入质押与罚没机制（Staking & Slashing）。公证节点若被审计发现签发伪造证据，将面临经济损失与永久禁抵，确保系统在博弈论层面保持诚实。
6.  **生态演进：从中心化控制到 DAO 治理 (Governance Evolution)**：
    *   **治理权下放**：公证节点的名录、准入标准以及信誉评分从中心化服务器迁移至链上 **DAO 合约**。
    *   **第三方服务准入 (Permissionless Services)**：任何第三方机构若希望建立自己的见证服务，均可根据 DAO 合约中的授信列表，自由筛选其信任的公证节点集合并进行“二次背书（Service-level Signing）”。
    *   **社区自治与生态承载**：系统的“信任锚点”最终由 DAO 管理。这种架构实现了我们的服务器从“管理者”向“参与者”的身份转换，极大地释放了服务器负载，并将 TweetyCat 进化为一个去中心化的、由社区共建共管的社交事实预言机协议。

## 6. 未来愿景：基于 OP Stack 的社交预言机 L3 (Future Roadmap)

TweetyCat 的终极形态将不再仅仅是一个社交工具，而是一个基于 **OP Stack** 开源堆栈、在 **Base (L2)** 之上构建的 **应用型三层网络 (L3)**。我们选择 OP Stack 作为底层框架，旨在通过“主场作战”实现性能与成本的极致优化，并深度集成到以太坊超级链 (Superchain) 生态。

### 6.1 技术选型：为什么选择 OP Stack？
*   **原生兼容性**：由于 Base 链本身基于 OP Stack 构建，TweetCat L3 采用同构协议栈可实现与 Base 资产与数据的秒级无感互通。
*   **安全继承**：L3 链的安全性通过 Base 间接锚定至以太坊 L1，确保了社交事实账本具备工业级的抗篡改能力。
*   **协议级集成**：通过修改 OP Stack 的执行引擎 (Execution Engine)，我们将 **ATA 验证逻辑** 固化在区块生产的共识层，使社交见证成为链的原生行为。

### 6.2 PoSA (Proof of Social Attestation) 共识机制
借鉴以太坊 PoS 机制，我们提出 **社交见证证明 (PoSA)** 共识：
*   **节点质押 (Staking)**：Miner App (小红) 需通过抵押项目代币成为“公证验证者”。这为其签发的每一份证明提供了经济学背书。
*   **罚没机制 (Slashing)**：若系统审计发现验证者签发了与原始流镜像不符的虚假证明，其质押资产将被原子化罚没。
*   **随机委员会 (Random Committees)**：每一笔社交事实验证由信令层随机分派多个见证节点。只有超过 2/3 的节点完成签名，该事实才会被合并打包。

### 6.3 价值驱动的新型挖矿模式与资产结算 (Value-Driven Mining & Asset Liquidity)
系统通过“双层资产模型”实现极速结算与真实价值的锚定：
*   **行为即挖矿 (Action-as-Mining)**：用户的真实社交活跃作为“社交算力”，在 L3 上直接开采出原生激励代币。这一过程在 L3 内部高频执行，确保零延迟的获得感。
*   **中继即安全 (Relay-as-Security)**：公证节点通过贡献带宽与计算资源，在 L3 上赚取区块奖励与服务费分润。奖励以 L3 原生资产结算，极大降低了大规模、高频次见证的清算成本。
*   **资产承兑闭环 (Value Anchoring Flow)**：
    1.  **资产注入**：广告主在 L2 (Base) 质押 USDC 或 ETH 至协议金库作为任务背书。
    2.  **L3 循环**：任务验证与节点激励在 L3 内部以原生资产形式高频化、零摩擦执行。
    3.  **跨层兑付**：用户或节点可随时将其持有的 L3 资产通过跨链路由，兑换并提现为 L2 金库中的硬通货。该过程利用 **x402 协议** 实现 Gas 抽象，确保兑付体验丝滑。

### 6.4 社交信用与 ID 体系 (On-chain Social Identity)
*   **抗女巫攻击 (Anti-Sybil)**：由于 ATA 协议要求真实的浏览器上下文和物理 IP 镜像，这天然筛选掉了流水线生成的僵尸账号，构建出一套真实的链上社交信用评分体系。
*   **L3 优势与气费抽象 (Gas Abstraction)**：作为 L3，TweetCat Chain 拥有独立的 Gas 逻辑。通过内置的“赞助商金库”，信誉良好的活跃用户或执行特定商业任务的用户可享受由广告主或协议直接赞助的**零 Gas 交易**。只有在资产向 Base 链跨层转移时，才通过 x402 协议进行中继结算，彻底抹平了 Web2 与 Web3 之间的用户摩擦。

### 6.4 社交智能合约 (Social Smart Contract - SSC)
为了解决社交行为多样性与区块链逻辑单调性之间的矛盾，L3 引入了 SSC 体系：
*   **任务即参数化合约**：每一个社交广告活动被定义为一组 **Schema（数据模式）**。广告主通过部署 SSC，将“验证文本偏移量”、“目标指纹”和“激励曲线”逻辑原子化。
*   **动态效果追踪**：利用 L3 的极低成本，SSC 可以实现“多步验证（如保持关注 7 天）”或“阶梯式归因奖励”，将复杂的广告商业审计逻辑完全转化为透明的 EVM 字节码运行。

### 6.5 三元经济映射模型 (The Triple-Actor Mapping)
我们将商业角色深度映射入 L3 共识层：
1.  **广告主 -> 资本泵浦 (Capital Pump)**：其支付行为被映射为合约的 Liquidity Injection (流动性注入)。他们不仅是任务的发起者，也是 L3 网络 Gas 的最终买单人。
2.  **用户 -> 社交算力 (Social Hashrate)**：用户的每一次点击和交互，通过 ATA 协议转化为一份份“工作量证明”。这种算力源于真实的社会活动，不可伪造且自带价值属性。
3.  **见证验证者 -> 状态公证人 (State Notaries)**：他们将 Web2 流量镜像转化为链上确定的 State Root。他们的“挖矿”过程就是对社会事实进行去中心化审计的过程。

## 补充方案：高价值广告任务的 Evidence 存证（条件可信）

### 目标与阶段划分

我们将“奖励发放证明”拆成两个阶段：

1) **当前阶段（本期实现）**：不做服务器签名、不做多客户端投票，只将任务相关的 **Evidence（证据摘要）** 写入数据库，供广告主后续审计与抽查。
2) **后续阶段（规划）**：服务器对任务生成签名并下发给随机挑选的浏览器客户端做二次验证；当随机抽取的 3 个客户端中有至少 2 个确认任务真实，则任务最终确认为真实。

### Evidence 存证的基本规则

- **只对高价值动作生成**：例如与奖励发放直接相关的关键任务（claim/withdraw/关键行为核验等）。
- **一任务一次**：同一个广告任务只允许生成一次 Evidence，多次提交视为无效（通过数据库唯一约束/幂等逻辑保证“最多一次生效”）。
- **最小化存储**：只保存“部分响应字段 + 哈希”，避免保存完整响应内容带来的隐私与存储成本。

### Evidence 建议包含的字段（示例）

以下字段用于证明“扩展在某时刻观察到某次请求/响应，并将其以不可篡改方式提交到服务器存证”：

- **任务标识**：`task_id`（或 `claim_id`）、`ad_id`、`b_x_id`、`a_x_id`
- **请求绑定（canonical）**：
  - `method`
  - `htu = origin + pathname`（不含 query/fragment）
  - `request_body_sha256`（如需）
- **响应摘要**：
  - `status`
  - `response_body_sha256`
  - `response_fields`（仅保存与任务判断直接相关的少量字段）
- **设备侧不可抵赖字段**：
  - `iat`（秒级时间戳）
  - `jti`（一次性随机 ID）
  - `device_jkt`（服务端从绑定的 `device_pubkey_spki` 计算得到的 thumbprint）
  - `device_signature`（Device Key 对 canonical request/response 摘要签名）

### 为什么“扩展观察到的 x.com 返回结果”在很大程度上是可信的（条件可信）

在 **浏览器与电脑本身没有被攻克**、扩展代码未被篡改的前提下，我们通过以下机制把“刷奖励/伪造证据”的成本抬到很高：

1) **一设备一密钥（Device Key）**：关键证据必须由本地不可导出的 Device Key 签名，服务器只接受与用户绑定的设备公钥验签通过的 Evidence。
2) **强制验签的关键 API**：服务器对高价值写接口强制校验签名输入（canonical request + bodyHash），杜绝“随便构造 HTTP 请求就能伪造证据”的情况。
3) **抗重放（jti 去重 + iat 时间窗）**：
   - `iat` 必须在允许窗口内（过期/未来漂移直接拒绝）
   - 同一设备在短 TTL 窗口内的 `(device_jkt, jti)` 只能生效一次（重复直接拒绝）
   这能有效抵御录包重放与并发重放。
4) **Coinbase 社交账号强绑定**：Evidence 与已登录态的用户身份绑定；攻击者想要刷奖励，必须在真实浏览器环境中安装正式扩展并完成账号登录/绑定，再满足上述签名与防重放约束。

在满足上述条件后仍要“刷奖励”，将需要付出极高的人力与环境成本（真实浏览器 + 正式扩展 + 登录绑定 + 通过签名与防重放约束）。对于这种“高成本、近似终端已被控制/人为操作”的场景，我们将其视为超出当前系统安全目标的范围（后续阶段会引入随机客户端复核进一步抬高成本与提高可审计性）。
